# HANDOFF: Платёжный Компас

## 1. Цель проекта

«Платёжный Компас» — Telegram-бот для финансовой дисциплины. Пользователь добавляет доходы, обязательные платежи, накопления и минимум на жизнь, а бот помогает понять:

- сколько нужно отложить на ближайшие платежи;
- сколько можно тратить;
- хватает ли денег до следующего дохода;
- есть ли риск просрочки;
- как предполагаемая покупка повлияет на план.

Главная продуктовая ценность: после каждого полученного дохода пользователь сразу видит реально свободную сумму.

## 3. Текущий статус

Финансовое ядро было доработано под упрощённую MVP-модель:

- отдельная таблица плановых распределений не добавлялась;
- основной источник правды по «Уже отложено» — `reserve_transactions`;
- если бот рассчитал, что с дохода нужно отложить сумму на платёж, эта сумма считается зарезервированной в плане;
- просмотровые разделы не должны создавать новые резервы;

- добавление и просмотр доходов;
- добавление и просмотр ближайших платежей;

## 5. Архитектура, которую важно сохранить

Слои:

```text
Telegram handlers
-> services
-> repositories
-> database
```

Правила:

- handlers только принимают ввод, управляют FSM, вызывают services/formatters и отправляют ответы;
- в handlers не должно быть SQL и бизнес-расчётов;
- расчёты держать в `app/calculations.py`;
- продуктовые сценарии держать в `app/services/`;
- работа с БД — в `app/repositories/`;
- пользовательские тексты — в `app/formatters.py` и `app/texts.py`;
- FSM использовать только для пошагового сбора данных;
- суммы хранить в копейках как `int`;
- «Уже отложено» не хранить в `obligations`, а считать по `reserve_transactions`;
- `reserve` и `manual_adjustment` увеличивают сумму, `release` уменьшает;
- просмотровые сценарии не должны менять БД.

## 6. Последние важные изменения

### `app/services/planning.py`

Добавлен единый сервис даты и горизонта планирования:

- `get_today()` возвращает текущую локальную дату бота;
- `get_now()` возвращает текущий локальный `datetime`;
- `get_planning_horizon_days(...)` берёт горизонт из настроек;
- `get_planning_horizon_end(...)` считает `base_date + planning_horizon_days`.

Добавлена dev-only настройка:

```env
CURRENT_DATE_OVERRIDE=YYYY-MM-DD
```

Она работает только при `DEV_MODE=true`. При `DEV_MODE=false` override игнорируется, даже если переменная заполнена.

При `PLANNING_HORIZON_DAYS=90` контрольные значения:

- `CURRENT_DATE_OVERRIDE=2026-05-30` даёт `horizon_end=2026-08-28`;
- `CURRENT_DATE_OVERRIDE=2026-06-10` даёт `horizon_end=2026-09-08`;
- `CURRENT_DATE_OVERRIDE=2026-06-20` даёт `horizon_end=2026-09-18`.

На этот сервис переведены пользовательские расчёты и debug-сценарии: «Ближайшие платежи», `/spend`, обработка дохода, cashflow gap check, `/debug_reserves`, `/debug_obligations`, income/return/dev handlers и scheduler.

### `app/models.py`

В `ReserveTransaction` добавлено поле `source`.

Смысл:

- `auto_plan` — автоматический резерв, созданный ботом при обработке дохода;
- `manual` — ручная корректировка «Уже отложено»;
- `release` или `auto_plan` на release-записи — отмена/уменьшение ранее созданного резерва.

Пользователю эти технические значения не показываются.

### `app/database.py`

Добавлена безопасная совместимость для старой SQLite-базы:

- при старте проверяется наличие колонки `source` в `reserve_transactions`;
- если колонки нет, выполняется безопасный `ALTER TABLE`.

Alembic не используется.

### `app/repositories/reserves.py`

Репозиторий резервов переписан вокруг жизненного цикла `reserve_transactions`:

- `create(...)` делает `add` и `flush`, но не делает `commit`;
- commit теперь находится в service layer;
- `sum_reserved_for_obligation(...)` считает общий резерв по платежу без фильтра по доходу;
- `has_auto_reserves_for_income(...)` определяет, есть ли активный auto-plan резерв по доходу;
- `sum_auto_reserved_for_income(...)` считает auto-plan резерв конкретного дохода с учётом release;
- `auto_reserved_by_obligation_for_income(...)` собирает auto-plan суммы по платежам для дохода;
- `release_auto_reserves_for_income(...)` создаёт release-записи вместо физического удаления старых резервов;
- `list_by_user(...)` используется для скрытого debug-экрана.

### `app/services/allocation.py`

Главное исправление финансового ядра:

- `process_received_income(...)` — единственная функция, которая создаёт автоматические резервы;
- `create_reserves_safely(...)` перед созданием каждой записи заново читает из БД актуальные:
  - обязательство;
  - уже отложенную сумму;
  - оплачено за период;
  - актуальный остаток;
- новая сумма резерва ограничивается актуальным остатком платежа;
- повторная обработка одного income не создаёт дубли;
- закрытые платежи не попадают в итоговое «Куда отложить»;
- итоговый `AllocationResult` строится по фактически созданным или уже существующим резервам;
- `recalculate_last_income(...)` и `recalculate_user_plan(...)` больше не создают `reserve_transactions`;
- добавлены info/debug-логи для диагностики резервирования;
- `release_auto_reserves_for_income(...)` отменяет auto-plan резервы дохода через release-записи.

### `app/services/obligations.py`

Доработаны ближайшие платежи и ручная корректировка:

- «Уже отложено» берётся через общий баланс резервов по платежу;
- отображаемое «Уже отложено» ограничивается суммой платежа минус оплачено;
- остаток считается через общую функцию расчёта;
- ручное увеличение создаёт `manual_adjustment` с `source="manual"`;
- ручное уменьшение создаёт `release` с `source="manual"`;
- после ручной корректировки делается commit.

### `app/services/incomes.py`

Доработан жизненный цикл дохода:

- при удалении received-дохода вызывается отмена его auto-plan резервов;
- при изменении суммы или даты received-дохода старые auto-plan резервы отменяются, затем доход обрабатывается заново;
- при смене статуса на received вызывается обработка дохода;
- при смене received на expected/cancelled вызывается release резервов.

### `app/services/savings.py`

Добавлен read-only preview для копилки:

- `preview_savings_for_income(...)` считает отображение без создания новых `savings_transactions`;
- это нужно, чтобы просмотровые разделы не меняли БД.

### `app/services/living_minimum.py`

Добавлен read-only preview:

### `app/services/financial_status.py`, `app/services/salary_plan.py`, `app/services/what_if.py`

Переведены на read-only preview там, где это важно для просмотровых сценариев.

### `app/handlers/common.py`

## 10. Ручной smoke-test финансового ядра

1. Убедиться, что по тестовым данным нет старых некорректных резервов.
2. Добавить платежи:
   - платёж на 5 600 рублей;

3. Добавить первый received-доход.

5. Проверить «Ближайшие платежи»: «Уже отложено» должно совпадать с резервами.
6. Добавить второй received-доход.
7. Платёж, закрытый первым доходом, не должен получить новый резерв.
8. Частично закрытый платёж должен получить максимум остаток.
9. «Уже отложено» не должно превышать сумму платежа.
10. Открыть «Сколько можно тратить», «Финансовый статус», «План до зарплаты», «Что если купить?» и снова проверить `/debug_reserves`: новых reserve-записей от просмотра быть не должно.

## 11. Важные ограничения

Не делать:

- не добавлять отдельную таблицу плановых распределений;
- не удалять базу;
- не удалять старые reserve-транзакции автоматически;
- не добавлять команду массового исправления резервов;
- не создавать резервы в просмотровых разделах;
- не переносить бизнес-логику в handlers;
- не показывать пользователю технические `source` и `transaction_type`.

Сохранять MVP-модель:

- рекомендовано отложить ботом = зарезервировано в плане;
- если пользователь фактически отложил другую сумму, он правит «Уже отложено» вручную.
