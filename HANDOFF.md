# HANDOFF: Платёжный Компас

## 1. Цель проекта

«Платёжный Компас» — Telegram-бот для финансовой дисциплины. Пользователь добавляет доходы, обязательные платежи, накопления и минимум на жизнь, а бот помогает понять:

- сколько нужно отложить на ближайшие платежи;
- сколько можно тратить;
- хватает ли денег до следующего дохода;
- есть ли риск просрочки;
- как покупка повлияет на план.

Главная продуктовая ценность: после каждого полученного дохода пользователь сразу видит реально свободную сумму.

## 2. Где продолжать работу

Продолжать нужно в папке, где лежит этот файл `HANDOFF.md`. В последней сессии рядом была другая папка с похожим названием и минимальной заготовкой проекта. Её не использовать: в ней нет полной архитектуры, сервисов резервирования и тестов.

## 5. Архитектурные решения, которые важно сохранить

Слои:

```text
Telegram handlers
-> services
-> repositories
-> database
```

Правила:

- handlers только принимают ввод, управляют FSM, вызывают services/formatters и отправляют ответы;
- в handlers не должно быть SQL и бизнес-расчётов;
- расчёты держать в `app/calculations.py`;
- продуктовые сценарии держать в `app/services/`;
- работа с БД — в `app/repositories/`;
- пользовательские тексты — в `app/formatters.py` и `app/texts.py`;
- FSM использовать только для пошагового сбора данных;
- суммы хранить в копейках как `int`;
- «Уже отложено» не хранить в `obligations`, а считать по `reserve_transactions`;
- резервы считать так: `reserve` и `manual_adjustment` как плюс, `release` как минус.

## 6. Изменённые файлы и смысл изменений

### `app/calculations.py`

Дорабатывалась чистая логика расчётов:

- расчёт остатка платежа с учётом уже отложенного и оплаченного;
- расчёт баланса резервов по списку транзакций;
- ограничение новой суммы резерва актуальным остатком платежа;
- корректировка ручного изменения «Уже отложено»;
- исключение полностью закрытых платежей из рекомендаций «Куда отложить».

Нужно перепроверить файл на повреждённые русские строки и прогнать тесты.

### `app/repositories/reserves.py`

Дорабатывалась работа с `reserve_transactions`:

- `create(...)` добавляет запись, делает `flush`/`refresh`, но не делает `commit`;
- `sum_reserved_for_obligation(...)` считает общий резерв по платежу без фильтра по доходу;
- `has_reserves_for_income(...)` должен проверять только реальные reserve-записи по доходу, где есть платеж, сумма больше нуля и тип именно `reserve`;
- добавлены функции чтения реальных резервов по доходу и всех резервов пользователя.

Важно: из-за переноса `commit` в service layer нужно проверить все места, где вызывается `reserves_repo.create(...)`.

### `app/repositories/payments.py`

Дорабатывался подсчёт оплаченной суммы за текущий платёжный период, чтобы старые оплаты не ломали расчёт остатка.

### `app/services/obligations.py`

Дорабатывался summary ближайших платежей:

- «Уже отложено» берётся через общий расчёт резервов;
- оплаченная сумма берётся по текущему периоду;
- отображаемое «Уже отложено» ограничивается суммой платежа минус оплачено;
- остаток считается общей функцией расчёта.

Нужно проверить commit после создания начального резерва при добавлении платежа и после ручного изменения «Уже отложено».

### `app/services/allocation.py`

Главное место последнего исправления:

- добавлены диагностические логи для процесса резервирования;
- расчёт перед записью получает актуальные резервы и оплаты из БД;
- новая запись резерва должна создаваться только на актуальный остаток;
- повторная обработка одного и того же дохода не должна создавать дубли;
- пересчёты для просмотра статуса не должны создавать reserve-транзакции;
- итоговое сообщение после полученного дохода должно строиться по фактически созданным резервам.

Нужно завершить проверку этого файла: убедиться, что запись реально коммитится, а потом читается в «Ближайших платежах».

### `tests/test_calculations.py`

Добавлялись тесты на:

- частично закрытый резервом платёж;
- полностью закрытый резервом платёж;
- частично оплаченный платёж;
- уменьшение резерва через release;
- второй доход, который должен резервировать только остаток;
- ограничение создаваемого резерва актуальным остатком.

Нужно запустить после восстановления Python.

### `tests/test_product_name.py`

Добавлен тест, который не должен позволять вернуть старое название продукта.

Важно: не добавлять в новые markdown-файлы старые технические и пользовательские названия, иначе тест может упасть.

## 10. Ручной smoke-test для текущего критического бага

После ручной очистки некорректных старых резервов в БД:

1. Добавить полученный доход на крупную сумму.
2. Убедиться, что бот показывает «Куда отложить».
3. Выполнить скрытую команду `/debug_reserves`.
4. Убедиться, что reserve-транзакции реально появились.
5. Открыть «Ближайшие платежи».
6. Убедиться, что «Уже отложено» совпадает с созданными резервами.
7. Добавить второй полученный доход.
8. Убедиться, что закрытый платёж не попадает в «Куда отложить».
9. Убедиться, что частично закрытый платёж получает максимум остаток.
10. Убедиться, что «Уже отложено» не становится больше суммы платежа.
